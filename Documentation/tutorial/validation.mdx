We left chapter 1 with a working slice and an honest problem: a librarian can register an author with a blank name, or the same author twice, and nothing stops them. A trustworthy app has to be able to say *no* — clearly, and as early as possible.

There are two different kinds of "no" here, and Arc handles them in two different places:

- **"That input is malformed."** A blank name is wrong on its face — you don't need to look at anything else to know it. That's a *validation* rule.
- **"That input collides with what already exists."** A duplicate name is only wrong relative to the rest of the system. That's a *business rule*, and it needs to consult state.

Let's add both.

## Validate the value, once

Your first instinct might be to validate `Name` inside a `RegisterAuthor` validator. But think about it: *every* command that takes an `AuthorName` wants the same guarantee — never blank. Arc lets you state that rule **on the value type itself**, so it's enforced everywhere an `AuthorName` is used, and you never write it twice.

<Steps>

1. **Write a `ConceptValidator` for the value type.** It reaches into the concept's underlying `Value` and applies ordinary [FluentValidation](https://docs.fluentvalidation.net/) rules:

   ```csharp
   public class AuthorNameValidator : ConceptValidator<AuthorName>
   {
       public AuthorNameValidator()
       {
           RuleFor(x => x.Value).NotEmpty().WithMessage("An author needs a name.");
       }
   }
   ```

   That's the whole change. You don't register it anywhere — Arc discovers validators by convention. From now on, any command carrying an `AuthorName` rejects a blank one before `Handle()` ever runs.

</Steps>

<Aside type="note" title="When the rule belongs to one command">
A `ConceptValidator` is right when the rule is true of the *value* everywhere. When a rule is specific to one command — "these two fields can't both be empty" — write a `CommandValidator<RegisterAuthor>` instead and use `RuleFor(c => c.SomeField)`. Same FluentValidation API, narrower scope.
</Aside>

## Reject the duplicate at the boundary

Uniqueness is the harder kind of "no", because it depends on what's already happened. You might be tempted to query the authors list first and check — but read models are eventually consistent, so two registrations racing at the same moment could both pass that check. The reliable place to decide is **inside `Handle()`**, against state Arc hands you.

Here's the move: a tiny read model that records which names are taken, and the command asks for *its own* name's entry. If Arc finds one, the name is a duplicate.

<Steps>

1. **Project the taken names.** A read model keyed by the name itself — one row per registered author name:

   ```csharp
   [ReadModel]
   [FromEvent<AuthorRegistered>]
   public record RegisteredAuthorName([Key] AuthorName Name);
   ```

2. **Let the command consult it.** Add the read model as a parameter to `Handle()`. Arc resolves the instance whose key matches the command's `Name` — so a non-empty match means "already registered." Return a `ValidationResult.Error` to reject, or the event to proceed:

   ```csharp
   [Command]
   public record RegisterAuthor(AuthorId Id, AuthorName Name)
   {
       public Result<ValidationResult, AuthorRegistered> Handle(RegisteredAuthorName? existing)
       {
           if (existing is not null && existing.Name != AuthorName.NotSet)
           {
               return ValidationResult.Error("An author with that name is already registered.");
           }

           return new AuthorRegistered(Name);
       }
   }
   ```

</Steps>

<Aside type="tip" title="Why this is safe">
Arc injects `RegisteredAuthorName` from the authoritative state at the moment the command runs, not from an eventually-consistent copy — so the duplicate check and the append happen as one consistent decision. This is the *dynamic consistency boundary*: you ask only for the state this one command needs, and you get it at full strength.
</Aside>

## The UI already knows

Here's the part that feels like a gift. You didn't touch the React code from chapter 1 — but the form now validates.

When you wrote the `ConceptValidator`, the proxy generator extracted that rule into the generated TypeScript too, so the `CommandDialog` runs it **on the client** and disables the confirm button on a blank name — no round trip. And when the server rejects the duplicate, the `ValidationResult.Error` comes back through the same proxy and `CommandDialog` renders it against the form. Your message — *"An author with that name is already registered."* — shows up under the field, because the dialog wires command results to fields for you.

```tsx title="AddAuthor.tsx — unchanged from chapter 1"
<CommandDialog<RegisterAuthor> command={RegisterAuthor} title="Add author" okLabel="Add">
    <InputTextField<RegisterAuthor> value={i => i.name} title="Name" />
</CommandDialog>
```

One rule on the value type, one rule in the handler, and both ends of the app honor them — because the validation, like the types, is generated from one source.

## What you built

- An `AuthorNameValidator` that guards the **value type**, enforced everywhere an `AuthorName` is used.
- A uniqueness **business rule** the command checks against authoritative state inside `Handle()`, returning a typed `ValidationResult.Error`.
- A React form that surfaces both — client-side and server-side — with no extra frontend code.
